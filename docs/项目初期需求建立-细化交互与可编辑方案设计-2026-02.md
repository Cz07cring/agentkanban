# 项目初期需求建立：细化交互与可编辑方案设计（2026-02）

## 1. 目标

针对“项目建立初期容易返工”的问题，本次落地目标是：

1. 前置收集更完整的信息（选择题 + 自然语言）。
2. 保留 AI 候选方案，同时允许用户直接编辑方案。
3. 创建前给出缺失项提示并强校验，避免不完整信息进入执行链路。
4. 将访谈结果结构化保存，供后续执行与复盘。

---

## 2. 本次实现的关键能力

### 2.1 访谈更细化（结构化问题）

除原有目标/节奏/范围/风险外，新增：
- 自动化程度（半自动/全自动）
- 可接受复杂度（低/中/高）
- 成功指标偏好（效率/质量/成本）

### 2.2 方案可编辑（满足“选项不合适可自定义”）

对当前选中的方案，用户可直接编辑：
- 方案标题
- 方案周期
- 方案风险
- 方案摘要（自然语言）
- 验收标准（按行编辑）

> 这使得 A/B/C 不再是“只能选不能改”，而是“可选 + 可改”。

### 2.3 自然语言补充输入

提供独立“自然语言补充阐述”输入区，允许直接写：
- 目标
- 边界
- 限制
- 期望效果

该信息会进入自动生成的启动评审包与结构化 payload。

### 2.4 创建前门禁

创建前会检查并提示缺失项（例如：
- 需求原文
- 候选方案数量
- 已选方案
- 验收标准

未满足时按钮不可提交。

### 2.5 结构化持久化

前端会将访谈结果汇总为 `init_brief`，后端持久化保存，至少包括：
- 需求原文
- 选择题回答
- 候选方案（含编辑后的内容）
- 选中方案
- 自动生成文档
- 自然语言补充

---

## 3. 业务满足度评估（对照你的诉求）

- “尽可能详细”：**已满足**（问题维度和输入内容更细）。
- “不满足可自己编辑”：**已满足**（方案编辑区已加入）。
- “支持自然语言阐述”：**已满足**（新增自然语言补充区并持久化）。
- “减少返工”：**部分满足并显著改善**（创建前门禁 + 结构化沉淀）；后续可继续加自动回溯和模板推荐。

---

## 4. 后续可选增强（P1）

1. 方案版本比较（改前 vs 改后 diff）。
2. 一键“基于自然语言补充重新生成方案”。
3. 质量评分矩阵（成本/风险/速度）自动打分。
4. 复盘回写（交付后对比 init_brief 与实际结果差异）。
